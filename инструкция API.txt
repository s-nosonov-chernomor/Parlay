Инструкция по API mqtt-bridge для фронта (порядок работы)
0) Базовые вещи

Base URL: http://<ip>:8000

Формат времени: ISO-8601, tz-aware (пример: 2026-01-29T00:00:00+03:00)

Авторизация:

Для опасных/управляющих и экспорта: заголовок x-api-token: <TOKEN>

Для “читать” (snapshot/stream/last/history) — обычно без токена (зависит от настроек, но сейчас так).

1) Экран “Дашборд/страница” (основной поток)

Цель: получить готовую сетку карточек, их биндинги, текущие значения и потом получать обновления.

Шаг 1. Получить snapshot страницы (рекомендуется для первого рендера)

GET /v1/ui/page/{page}/snapshot

Пример:

GET /v1/ui/page/dashboard/snapshot

Ответ содержит:

elements[] — ui элементы (карточки зон), уже с полями и meta

bindings[] — биндинги (ui_id + bind_key -> topic + source + note)

states[] — состояние режима для ui (AUTO/MANUAL/…)

last[] — последние значения по топикам (ts/status/value)

Как фронту использовать

Рисуем сетку карточек по elements (позиции cz/row_n/col_n, заголовок title, мета zone_code).

Для каждой карточки показываем параметры по bindings:

bind_key — ключ параметра (то, что выбирает пользователь)

note — человекочитаемое название (если есть)

topic — куда подписываться/откуда брать last

Значения подтягиваем из last по ключу topic.

Это самый быстрый и “безопасный” способ стартовать страницу без кучи запросов.

Шаг 2. Подписаться на поток обновлений страницы

GET /v1/ui/page/{page}/stream (SSE)

Пример:

GET /v1/ui/page/dashboard/stream

Фронт слушает события и патчит значения в сторе по topic.

Рекомендации

Держать 1 SSE на страницу.

При переподключении передавать Last-Event-ID (если фронт хранит), чтобы минимизировать пропуски.

Если SSE отвалился — делаем backoff и после восстановления можно дернуть snapshot, чтобы выровняться.

2) Экран “Карточка зоны” (управление и режим)

У тебя ui_id выглядит так: zone:1:r10:c11.

Получить режим/состояние зоны (обычно уже есть из snapshot)

Если нужно отдельно:
POST /v1/ui/{ui_id}/mode — это установка режима, не чтение.

Чтение состояния в целом делаем через:

snapshot страницы (см. выше), там есть states[].

Установка режима зоны

POST /v1/ui/{ui_id}/mode

Тело:

{
  "mode_requested": "AUTO",
  "schedule_id": "schedule_1"
}


mode_requested: WEB|AUTO|PRIVA|MANUAL

schedule_id обязателен для AUTO (по смыслу)

3) Отправка команд на зону (set по bind_key)

Фронт не должен знать topic напрямую, он работает через ui_id + bind_key.

POST /v1/ui/{ui_id}/set

Тело:

{
  "bind_key": "led.dim_b",
  "value": 80,
  "as_json": true,
  "requested_by": "frontend",
  "correlation_id": "uuid-optional"
}


Ответ вернёт:

фактический topic и payload

mode_effective (важно: команда могла быть принята, но режим может блокировать)

UI-паттерн

После отправки команды — оптимистично подсветить “отправлено”.

Фактическое применение подтверждать по SSE/last (когда поменялось значение/ts).

4) Запрос данных для графиков (Query API)

Это как раз твой query/run и query/export.xlsx.

4.1 Получить временной ряд (для графика/таблицы)

POST /v1/query/run

Тело:

{
  "ui_ids": ["zone:1:r10:c11"],
  "bind_keys": ["hps.enabled", "led.enabled", "elec.i_l1", "led.dim_b"],
  "start": "2026-01-28T00:00:00+03:00",
  "end": "2026-01-29T00:00:00+03:00",
  "bucket_s": 900,
  "limit": 200000
}


Важно:

bind_keys должны быть массивом строк, без запятых внутри строки.

Неправильно: "elec.i_l1, led.dim_b"

Правильно: ["elec.i_l1", "led.dim_b"]

Ответ:

rows[] — строки вида: ts, ui_id, zone_code, source_id, bind_key, note, topic, value_num/value_text

columns[] — порядок колонок

meta — статистика (сколько разрешилось биндингов и т.п.)

Как фронту работать с результатом

Группировать по серии: (ui_id, bind_key) или (ui_id, bind_key, note).

Для графиков использовать value_num (если null — значит текстовый параметр).

Если bucket_s задан, ряд уже агрегирован по времени (avg).

4.2 Экспорт в Excel

POST /v1/query/export.xlsx (требует x-api-token)

Заголовок: x-api-token: <TOKEN>

Тело запроса — то же самое, что /query/run

Ответ: бинарный файл Excel.
Фронт:

выставляет responseType: "blob"

сохраняет как файл (Content-Disposition приходит с именем)

5) “Справочники” и ручная навигация по топикам (если нужно)

Эти эндпоинты полезны для админки/диагностики, но для твоего основного фронта (карточки зон) обычно не обязательны.

Параметры-список

GET /v1/parameters?prefix=/Теплица/Досветка/&limit=5000

Параметры-дерево

GET /v1/parameters/tree?prefix=/Теплица/Досветка/&limit=20000

Последние значения (по prefix или списку topics)

GET /v1/readings/last?prefix=/Теплица/Досветка/ЩД32/

История по одному topic

GET /v1/readings/history?topic=<topic>&start=...&end=...&limit=5000

Stream readings (SSE) по topics/prefix

GET /v1/readings/stream?prefix=/Теплица/Досветка/

6) Щиты (cabinets) — экран щита и диагностика

Это отдельный раздел UI, полезен для “экран щитов”.

Список щитов

GET /v1/cabinets

Снимок щита (щит + его параметры + линии)

GET /v1/cabinets/{source_id}/snapshot?include_line_bindings=true

Биндинги щита

GET /v1/cabinets/{source_id}/bindings

Обновить/создать биндинг щита (админка)

POST /v1/cabinets/{source_id}/bindings + x-api-token

Здоровье (шахматка)

GET /v1/cabinets/health/grid

Детально здоровье щита

GET /v1/cabinets/health/{source_id}

Рекомендованная последовательность для фронта (коротко)

Первый рендер страницы

GET /v1/ui/page/dashboard/snapshot

Рисуем карточки + значения

Живые обновления

GET /v1/ui/page/dashboard/stream (SSE)

Патчим значения по topic

Управление

POST /v1/ui/{ui_id}/set (команды)

POST /v1/ui/{ui_id}/mode (режим)

Графики

POST /v1/query/run (получить ряды)

POST /v1/query/export.xlsx (скачать Excel)

Практические рекомендации (чтобы не словить “пустые rows”)

Всегда перед query/run фронт берет список доступных bind_key для выбранных ui_id из snapshot (bindings[]), и предлагает выбирать только их.

Если пользователь вводит bind_key руками — валидировать: строка без запятых, без пробелов вокруг, только один ключ.

Если query/run вернул meta.resolved_bindings < requested_ui_ids * bind_keys — это не ошибка, значит часть параметров не привязана к этой зоне/щиту.
